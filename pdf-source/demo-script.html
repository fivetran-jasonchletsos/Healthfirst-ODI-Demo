Healthfirst · Fivetran ODI Demo

Demo script — Mutual Evaluation Plan walkthrough

Live demo: fivetran-jasonchletsos.github.io/Healthfirst-ODI-Demo  ·  Repo: github.com/fivetran-jasonchletsos/Healthfirst-ODI-Demo  ·  Prepared for: Ari Rassler, Niraj Vora

Three use cases from Healthfirst's Mutual Evaluation Plan drive this demo: Salesforce (Cloud Application Ingestion), Oracle Database (Database data Ingestion), and S3/SharePoint (File data Ingestion) — all landing in a single AWS S3 Managed Data Lake destination. Every capability claim in the app is sourced from a real fivetran.com/docs page and labeled confirmed / inferred / not verified — use that discipline in the room: if an evaluator pushes on a number, you can show them exactly where it comes from.

Opening line
"Today, bringing on a new data source at Healthfirst means engineering builds and maintains a custom pipeline — that's the backlog problem in your evaluation plan. What we're showing you is one platform: three connectors, one governed lake, no per-source engineering lift."

2. Architecture — the MDLS story (spend the most time here)

Land on /architecture. This is the page Ari asked us to sell — the AWS S3 Managed Data Lake destination is the centerpiece, not a footnote.

  • Healthfirst owns the bucket. Fivetran manages table structure, metadata, catalog, and maintenance; the data itself lives in Healthfirst's own S3 bucket the whole time.
  • One copy, many engines. Apache Iceberg tables + AWS Glue Data Catalog (Fivetran keeps both in sync every run) mean Athena, Snowflake, Redshift, and EMR all read the same tables — no per-tool rebuild, no waiting on the EDM backlog to stand up a new analysis.
  • Copy-on-Write. Updates/deletes write a new Parquet file excluding the changed row rather than editing in place — query engines always see correct, de-duplicated data.
  • Automated table maintenance. Snapshot expiration, orphan-file cleanup (every other week), metadata version retention — no customer-scheduled compaction jobs.
  • Validate and transform, downstream, on purpose. Fivetran is extract-and-load only (no pre-load cleansing beyond row filtering) — Great Expectations (GX Core, which Fivetran now stewards) validates bronze before promotion to silver, and dbt Wizard (dbt Labs' AI agent, real and shipping post the Fivetran+dbt Labs merger) authors the models that build silver/gold. This is a deliberate separation of concerns, not a gap.
If asked "why not just use Snowflake as the destination"
"You can — Snowflake reads these same Iceberg tables via a catalog-linked database. The point of MDLS is you're not choosing one warehouse to own the data; the data lives in your S3 bucket and every compatible engine reads it the same way."
Confirm before contract Partitioning strategy for the Iceberg/Delta tables (time/source/classification-based folder layout) is not publicly documented by Fivetran for this destination — get this in writing from the Fivetran team before making a partition-pruning performance promise to Healthfirst's engineers.

3. Sources — the receipts

Land on /sources. This is the page to open if an evaluator says "prove it." Every row has a linked source doc. Talking points, one per connector:

  • Salesforce: REST + Bulk API, incremental via timestamp fields; formula fields excluded from normal sync (captured separately, reconstructed downstream) — a real nuance, not a gap to hide.
  • Oracle: Binary Log Reader (log-based CDC on redo logs) is the method to lead with — LogMiner is sunset May 15, 2026, don't propose it. Capacity ceiling: 200 GB/hour redo volume, 5 TB aggregate table size.
  • S3 / SharePoint: Two sync modes each (Magic Folder = simple, full-file reload; Merge = pattern-based, incremental, primary-key strategy locked after first sync).

4. POC Success Criteria — scored on Healthfirst's own rubric

Land on /poc-criteria. Structured on the customer's own capability matrix (Core A.I–A.XIII, Value-Add B.I–B.V) — this is the actual scoring sheet, not our own checklist.

The layer badge is the whole point
Every requirement is tagged Fivetran / S3-Lake-Formation / Joint. Lead with this in the room: "we're not going to tell you Fivetran does something AWS actually does — here's exactly which layer owns each capability." Retention rules, legal holds, WORM storage, and tiered storage are honestly labeled as S3/Lake-Formation capabilities Fivetran writes into, not Fivetran features.
Items marked "not verified" — don't overclaim these live Resumable/multipart file uploads into MDLS specifically; structured per-record error quarantine with correlation IDs; exponential-backoff phrasing for ingestion retries (Fivetran's retry is tiered, not exponential, per its own docs). Say "let's confirm that with the Fivetran product team" rather than guessing.

5. Compliance — HIPAA leads, and say the shared-responsibility line out loud

Land on /compliance. Fivetran will sign a BAA and holds HITRUST i1 (tied to Business Critical plan), SOC 1/2, ISO 27001/27701, PCI DSS Level 1. Say this proactively: "Fivetran's own docs are explicit that HIPAA compliance is shared — HHS doesn't certify HIPAA compliance for anyone. A signed BAA and HITRUST are necessary, not sufficient; Healthfirst still owns PHI classification, minimum-necessary access, and non-production data hygiene." Saying this first builds more trust than an evaluator finding it themselves.
